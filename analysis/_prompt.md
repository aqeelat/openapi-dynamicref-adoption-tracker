# Tool DynamicRef Analysis Prompt

Use this prompt with a coding or research agent to analyze a tool from `TOOLING_CATALOG.md` and determine how likely it is that `$dynamicRef` / `$dynamicAnchor` support can be added and accepted upstream.

## Prompt

Research `<tool-name>` deeply for OpenAPI 3.1+ / JSON Schema 2020-12 `$dynamicRef` and `$dynamicAnchor` support.

Do not implement changes. The tool's repository is cloned at `/tmp/<tool-name>/`. Use it for source exploration: grep for keywords, read source files, check test structure, and verify dependency manifests. Fall back to web sources (GitHub web UI, package registries) for issues, PRs, release history, and content not on the default branch.

Return findings in the structure below.

## Required Questions

### Identity

- What is the tool's repository URL?
- What package(s) should be analyzed?
- Which category does it belong to: parser/resolver, validator, documentation renderer, API client, mock server, SDK generator, or spec producer?
- What version was analyzed?

### Maintenance And Landing Likelihood

- When was the most recent meaningful commit?
- When was the latest release or package publish?
- How many open issues and PRs are there, if visible?
- Who are the apparent maintainers or active reviewers?
- Are external PRs being merged recently?
- Does the project appear active, slow-moving, maintenance-only, or inactive?
- Estimate the likelihood that a well-scoped `$dynamicRef` PR would land: high, medium, low, or unknown.
- Explain the estimate with evidence, not vibes.

### Dependency Chain

- What parser/resolver/schema-renderer/validator packages does the tool use?
- Does it use `@apidevtools/json-schema-ref-parser`, a fork, Swagger ApiDOM, Redocly OpenAPI Core, AJV, `openapi-sampler`, `json-schema-tree`, or a custom resolver?
- Is `$dynamicRef` ignored because traversal only checks `$ref`?
- Does the tool bundle or dereference specs before rendering/generation?
- Verify the current dependency chain from package manifests and lockfiles. Do not rely on catalog notes; dependencies may have changed.
- If the repository moved or was renamed, record the current canonical repo and the legacy repo/name.
- If the tool advertises OpenAPI 3.1/3.2, verify whether it actually preserves JSON Schema 2020-12 dialect or silently downgrades to draft-07/OpenAPI 3.0-style behavior.
- Classify each upstream dependency: if the tool delegates `$dynamicRef` resolution to it, is that upstream **Correct** (record as "Backed by" in the catalog) or **Partial/No support** (record as "Blocked by")? State the evidence.

### Current DynamicRef Support

- For runnable tools (generators, validators, linters, parser-bundlers, importers, renderers), every behavioral claim below MUST cite an observed fixture result. Label any uncited claim as an inference and state what observation would confirm it. Do not infer behavior from source control flow when a fixture run is possible.
- Search source, tests, issues, and PRs for `$dynamicRef`, `$dynamicAnchor`, `$recursiveRef`, `$recursiveAnchor`, `$anchor`, `$id`, and OpenAPI 3.1/3.2.
- Does the tool parse/preserve the keywords?
- Does it semantically resolve dynamic scope?
- Does it generate samples/examples from dynamically resolved schemas?
- Does it validate request/response bodies using dynamic scope?
- If unsupported, what is the exact failure mode: crash, parse error, unresolved keyword display, empty schema, scalar fallback, `any`/`unknown`, wrong binding, infinite recursion, or silent degradation?
- Distinguish internal metaschema workarounds from user-document support. A tool may replace `$dynamicRef` in its own OpenAPI metaschema for validator compatibility while still not supporting user specs that contain `$dynamicRef`.
- Distinguish keyword recognition/rendering from semantic dynamic-scope resolution. Showing `$dynamicRef: #x` in a schema panel is not support if concrete operation schemas and examples remain unresolved.

### Implementation Plan

- **Generics are achievable. Do NOT default to "architecturally incompatible / materialize concrete types / docs only."** Orval proves real generic-type emission (`interface PaginatedTemplate<T>`) lands upstream (PR #3353). For generators/producers, read [`analysis/orval-reference.md`](orval-reference.md) and design a real generic-emission path modeled on Orval's two patterns. The "generics are erased" objection usually doesn't apply at the compile-time/reflection stage where producers operate.
- **Verify `$ref`-sibling preservation before concluding Pattern B is unsupported.** If the tool's parser drops `$ref` siblings (`$defs`, `$dynamicAnchor`, `$id`), Pattern B is *unimplementable* on that parser — not "not implemented." Check the catalog's §1 "Sibling-safe" column. See [`analysis/ref-sibling-preservation.md`](ref-sibling-preservation.md) + IMPLEMENTATION_GUIDE.md "Step 0" for the probe.
- What is the smallest useful change?
- Should the first PR preserve/pass through `$dynamicRef` or implement semantic dynamic-scope resolution?
- Where should dynamic-scope resolution live: parser, schema tree, sample generator, UI component, validator, or producer schema model?
- What tests should accompany the change?
- What backwards-compatibility risks exist?
- What behavior should remain unchanged for ordinary `$ref`, `$id`, `$anchor`, circular refs, and OpenAPI 3.0 documents?
- If the project spans multiple packages, identify the lowest shared layer where support belongs. Avoid UI-only fixes when schema rendering and sample generation consume a shared resolver.
- If full dynamic-scope resolution is too large, identify a safe first PR such as pass-through preservation or explicit unresolved `$dynamicRef` fallback display.

### Recommendation

- Should this tool be tackled now, later, or skipped?
- If now, should the work start with an issue, a failing test PR, or an implementation PR?
- What is the expected upstream acceptance likelihood?
- What downstream tools would benefit if this lands?
