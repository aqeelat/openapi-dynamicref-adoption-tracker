# openapi-fetch

## Summary

`openapi-fetch` is the type-safe fetch client sibling of `openapi-typescript` (same monorepo, `openapi-ts`). It does **no schema parsing or `$ref`/`$dynamicRef` resolution of its own** — its sole runtime dependency is `openapi-typescript-helpers`, and it is consumed by importing the `.d.ts` types that **`openapi-typescript`** emits. Therefore its `$dynamicRef` behavior is entirely inherited from openapi-typescript: `$dynamicRef` slots degrade to `unknown` in the types, and so the fetch client types them as `unknown`. No crash, no warning. The fix lives entirely in openapi-typescript (issue openapi-ts#2029, maintainer buy-in). openapi-fetch inherits the fix for free once openapi-typescript lands it.

## Dependency Chain

```
openapi-fetch
  → openapi-typescript-helpers   (the only runtime dep; no spec parser, no resolver)
  → (at author-time) types generated by openapi-typescript, which uses @redocly/openapi-core
```

- openapi-fetch ships **no** JSON Schema / OpenAPI parser, no `$ref` resolver, no AJV, no `@apidevtools/json-schema-ref-parser`. It is a thin generic wrapper over `fetch` parameterized by generated types.
- The dynamicRef gap is therefore **not openapi-fetch's** — it is `openapi-typescript`'s type-emission gap. **Blocked by** openapi-typescript.

## Current DynamicRef Behavior

Verified by running `openapi-typescript` on the fixtures (which is what an openapi-fetch consumer does at author-time):

| Fixture | `$dynamicRef` slot | Emitted type (what openapi-fetch sees) |
|---|---|---|
| generic-schema-binding | `PaginatedTemplate.items` | `unknown[]` (should be `User[]`/`Group[]`) |
| recursive-category-tree | `BaseCategory.children` | `unknown[]` (should be the concrete category type) |

Because openapi-fetch keys its request/response types off the generated `paths`/`components["schemas"]` interfaces, a `$dynamicRef`-bearing response body is typed as `unknown` end-to-end. No runtime effect (fetch still works); pure type loss.
